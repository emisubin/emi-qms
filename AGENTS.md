# AGENTS.md — EMI QMS 개발 규칙

## 1. 제품 목적

이 저장소는 EMI의 프로젝트·제품·판넬을 QR로 식별하고 설계, 자재, 생산, 품질, 부적합 조치, 포장, 출하, 납품 확인까지 추적하는 내부 QMS/MES 경량 통합 웹시스템이다.

## 2. 작업 전 필수 절차

1. 작업 전에 `README.md`, 관련 `docs/`, 관련 ADR, 해당 `tasks/` 문서를 읽는다.
2. 업무 규칙이 불명확하거나 문서가 충돌하면 코드를 추측해 작성하지 말고 충돌 내용을 명시한다.
3. 한 작업에서는 하나의 기능 또는 하나의 명확한 기술 목적만 변경한다.
4. 기존 공개 API, DB 스키마, 상태 전이를 바꾸는 경우 영향 범위를 먼저 정리한다.
5. 구현 전 현재 테스트를 실행하고, 구현 후 관련 테스트를 추가·실행한다.

## 3. 고정 업무 원칙

- 최상위 추적 단위는 프로젝트이며, 실제 진행·검사·출하 단위는 개별 제품·판넬이다.
- 제품당 QR은 하나만 발급한다. QR에는 민감정보를 직접 넣지 않고 불투명한 식별자 또는 토큰만 사용한다.
- 같은 QR을 스캔해도 사용자 역할과 제품 상태에 따라 기본 화면과 허용 작업이 달라진다.
- 권한은 UI 숨김으로 끝내지 않고 모든 API에서 서버가 검증한다.
- 승인·완료된 기록은 직접 덮어쓰거나 삭제하지 않는다. 정정 요청, 사유, 승인자, 변경 전·후 값을 저장한다.
- 최초 검사, 조치, 재검사 결과는 서로 다른 이력으로 저장한다.
- 열린 부적합이 하나라도 있으면 품질 승인과 출하를 차단한다.
- 프로젝트 상태는 가능하면 하위 제품 상태에서 계산한다.
- 설계 도면·BOM·검사양식에는 Revision과 적용시점이 있어야 한다.
- PDF는 승인 시점 데이터의 불변 스냅샷으로 생성한다.

## 4. 예정 기술 규칙

기본 아키텍처는 ADR-0001을 따른다.

- Frontend: React + TypeScript, 반응형 웹
- Backend: ASP.NET Core Web API
- Database: PostgreSQL
- Authentication: Microsoft Entra ID
- File storage: Azure Blob Storage
- Local development: 컨테이너 기반 PostgreSQL 허용

새 프레임워크나 핵심 라이브러리를 도입하기 전에 필요성, 대안, 보안·유지보수 영향을 ADR로 기록한다.

## 5. 코드 품질

- TypeScript와 C#의 엄격한 타입검사를 사용한다.
- 도메인 규칙은 화면 컴포넌트나 Controller에 직접 흩어놓지 않는다.
- 의미 없는 약어와 매직 넘버를 금지한다.
- 오류를 무시하거나 빈 catch로 숨기지 않는다.
- 사용자에게는 내부 스택트레이스·DB 정보·파일 경로를 노출하지 않는다.
- 모든 날짜·시간은 서버에서 UTC로 저장하고 화면에서 사용자 표준시간대로 표시한다.
- 금액·수량·측정값의 단위와 반올림 규칙을 명시한다.

## 6. 데이터·보안

- 비밀번호, 연결문자열, API 키, 인증서를 코드·커밋·로그에 넣지 않는다.
- 실제 고객사·프로젝트·개인정보를 개발 및 자동 테스트 데이터로 사용하지 않는다.
- 파일 업로드는 크기, 확장자, 실제 콘텐츠 유형, 파일명을 검증한다.
- 파일 다운로드 시에도 프로젝트·제품 권한을 다시 검증한다.
- 사용자 입력은 서버에서 검증하고 DB 접근은 매개변수화한다.
- 로그인, 조회, 등록, 수정, 승인, 정정, 다운로드, 권한 변경을 감사 로그 대상으로 본다.
- 로그에 인증 토큰, 비밀번호, 개인정보 전체값을 남기지 않는다.

## 7. 테스트 원칙

기능 변경에는 최소한 다음 중 필요한 테스트를 포함한다.

- 도메인 단위 테스트
- API 통합 테스트
- 권한 거부 테스트
- 상태 전이 테스트
- DB 트랜잭션 롤백 테스트
- 주요 사용자 흐름 E2E 테스트

정상 경로만 검사하지 말고 무권한 접근, 중복 요청, 잘못된 상태 전이, 필수값 누락, 파일 업로드 실패를 포함한다.

## 8. DB 변경

- DB 변경은 마이그레이션으로 관리한다.
- 이미 배포된 마이그레이션을 수정하지 않는다.
- 파괴적 변경에는 데이터 이전 및 롤백 계획을 포함한다.
- 외래키, 유일성, 필수값 등 가능한 무결성 규칙은 DB에서도 보장한다.
