# 11. 확정 요구사항 기준선

이 문서는 TASK-002A 기준으로 확정된 EMI QMS 업무규칙을 정리합니다. 실제 프로젝트 등록 API, 프로젝트 DB 테이블 확장, 화면, QR 생성, 구매·제조·품질·알림 기능은 후속 TASK에서 구현합니다.

## 권한 원칙

- 모든 활성 사내 사용자는 모든 프로젝트를 조회할 수 있습니다.
- 모든 활성 사내 사용자는 모든 부서 기능을 조회할 수 있습니다.
- 등록·수정·완료·취소는 해당 업무 담당 부서만 가능합니다.
- UI 버튼 숨김은 보조 수단이며, 모든 쓰기 API는 서버에서 권한을 검사합니다.
- `Project.Read.All`이 있는 활성 사내 사용자는 모든 프로젝트를 조회할 수 있습니다.
- `Project.Read.All`이 없는 제한 계정은 `UserProjectAccess`에 배정된 프로젝트만 조회할 수 있습니다.
- `UserProjectAccess` 모델은 향후 외부 사용자 또는 제한 계정을 위해 유지합니다.
- 프로젝트 쓰기권한은 전체 조회권한과 별도로 검사합니다.
- legacy `projects.access.all`은 전체 프로젝트 조회를 허용하지 않으며, 새 Endpoint와 Policy는 `Project.Read.All`만 전체 조회권한으로 사용합니다.

## 민감정보

- 판매금액은 `Project.SalesAmount.Read` 권한이 있는 Sales와 System Administrator만 조회할 수 있습니다.
- 제조 소요시간은 `Manufacturing.WorkTime.Read` 권한이 있는 Sales와 System Administrator만 조회할 수 있습니다.
- Design, Procurement, Production Planning, Manufacturing, Quality, Logistics, Read Only는 위 민감정보 조회권한을 갖지 않습니다.
- 실제 Project DTO와 제조시간 API 응답 필터링은 해당 업무 API TASK에서 구현합니다.

## 역할 후속 조건

- TASK-003A 프로젝트 등록 기능은 Sales 권한으로 진행합니다.
- Design Role + `PanelInfo.Update`는 TASK-003B 설계 패널정보 기능 구현 전에 필수입니다.
- Procurement Role + `ProcurementPlan.Update`는 구매 기능 TASK 구현 전에 필수입니다.
- 이번 TASK-002A에서는 Design과 Procurement Role을 추가하지 않습니다.

## 프로젝트 등록

필수 입력값은 고객사, Item, PJT Code, PJT Title, 면수, 납기일, 영업담당자입니다. 선택 입력값은 판매금액, 통화, 납품장소입니다.

- PJT Code 중복은 허용합니다.
- PJT Title은 앞뒤 공백 제거와 대소문자 무시 기준으로 중복을 허용하지 않습니다.
- 내부 Project ID는 UUID입니다.
- 면수만큼 `P01`, `P02` 형식의 제품 Placeholder를 생성합니다.
- 제품명과 패널명은 최초 프로젝트 등록 때 입력하지 않습니다.
- 임시저장은 없습니다.
- 면수 변경, 프로젝트 보류·취소·해제는 영업만 가능합니다.
- 보류·취소 시 해당 프로젝트의 모든 진행 업무를 중단합니다.
- 보류·취소 상태에서도 기존 데이터는 조회할 수 있습니다.

## 설계

- 도면 업로드와 도면 Revision 관리는 현재 범위가 아닙니다.
- 설계 입력값은 패널명과 사이즈입니다.
- 직접 입력과 엑셀 업로드를 모두 지원합니다.
- 사이즈 기본 구조는 W, H, D, 단위 mm입니다.
- 패널명과 사이즈 완료 후 QR 생성 조건을 충족합니다.
- 실제 QR 생성은 후속 QR TASK에서 구현합니다.

## 구매

- 직접 입력과 엑셀 업로드를 지원합니다.
- 필드는 통상납기, 발주품목, 기술 담당자, 발주일, 입고예정일입니다.
- 실제 입고완료 여부는 현재 관리하지 않습니다.
- 미입고, 부분입고, 입고지연 상태를 사용하지 않습니다.
- 입고예정일과 현재 날짜 차이만 표시합니다.
- IQC 시작방식은 품질 TASK 전에 결정합니다.

## 제조

- 제조단계는 10개입니다.
- 단계 순서는 강제하지 않습니다.
- 한 제품에서 여러 제조단계가 동시에 진행될 수 있습니다.
- 각 단계에는 시작·종료 버튼이 있습니다.
- 종료 결과는 단계 완료 또는 작업 중단입니다.
- 단계 완료 시 사유는 필요하지 않고, 작업 중단 시 사유는 필수입니다.
- 시작 사용자와 종료 사용자가 달라도 됩니다.
- 모든 활성 프로젝트 제품을 통합 일괄 선택할 수 있습니다.
- 동일 제조단계만 하나의 작업 묶음으로 처리합니다.
- 최대 50면을 선택할 수 있습니다.
- 작업 묶음 저장을 지원합니다.
- QR은 매 동작마다 스캔하지 않고 업무 진입·대상 확인 수단으로 사용합니다.
- 제조 소요시간은 Sales와 System Administrator만 조회합니다.
- 다른 부서는 단계 상태와 작업 이력만 조회합니다.
- 정확한 Man-hour는 현재 계산하지 않습니다.

## 품질

검사 종류는 IQC, LQC, 자체검수, 고객검수, FAT입니다.

- 모든 품질 체크리스트 항목에 사진 1장 이상이 필수입니다.
- 부적합 등록, 제조 부적합 조치 완료, 재검사 항목에도 사진이 필수입니다.
- 사진 누락 시 항목 완료와 품질 승인을 할 수 없습니다.
- 실제 체크리스트 항목은 품질 TASK 전에 확정합니다.

## QR

- QR 스캔 후 미로그인 사용자는 Microsoft 365 로그인 화면으로 이동합니다.
- 로그인 후 해당 제품과 사용자 역할을 확인합니다.
- 제조 역할은 제조 페이지, 품질 역할은 품질 페이지, 물류 역할은 물류 페이지로 이동합니다.
- 영업과 관리자는 제품 종합현황으로 이동합니다.
- 여러 역할 사용자는 활성 역할 또는 역할 선택 화면을 사용합니다.
- QR에는 고객사·프로젝트 정보를 직접 넣지 않고 임의 토큰을 사용합니다.
- 현재 작업에서는 Microsoft Entra ID와 실제 QR 생성을 구현하지 않습니다.

## 알림

- 프로젝트별 업무 담당자는 생산계획 단계에서 지정합니다.
- 단계 담당자가 있으면 해당 담당자가 알림을 받습니다.
- 담당자가 없으면 프로젝트 영업담당자가 받습니다.
- 영업담당자도 비활성 상태이면 시스템 관리자가 받습니다.
- 웹 내부 알림이 공식 원본입니다.
- Teams와 이메일은 보조 알림입니다.

## 진행률

- 화면에는 퍼센트만 표시하고 완료 수 / 전체 수는 숨깁니다.
- 내부적으로 적용된 체크리스트의 완료비율로 계산합니다.
- 면수가 늘어나면 제품별 단계가 분모에 추가됩니다.
- 완료 상태만 진행률에 반영하고, 진행 중·중단은 반영하지 않습니다.
- 취소 제품과 승인된 해당 없음 항목은 분모에서 제외합니다.
- 프로젝트 전체 완료조건 충족 전에는 최대 99%입니다.
- 모든 필수 항목 완료 시 100%입니다.

## 정정·되돌리기

- 각 데이터를 입력한 담당 부서 사용자는 수정·되돌리기가 가능합니다.
- 수정·되돌리기 시 상세 이력을 저장합니다.
- 저장 항목은 변경 대상, 변경 전 값, 변경 후 값, 수정자, 수정일시, 수정 사유입니다.
- 뒤 단계가 이미 시작된 경우의 영향은 각 TASK 전에 별도 결정합니다.

## PWA

- 반응형 웹과 PWA 방식으로 운영합니다.
- 공개 앱스토어 배포는 초기 범위가 아닙니다.
- 현재 작업에서는 Service Worker와 오프라인 캐시를 구현하지 않습니다.
