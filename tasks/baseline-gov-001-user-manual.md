# BASELINE-GOV-001 Task 종료 정책 User Manual

## 1. 이 정책이 필요한 이유

Task가 끝났다는 말은 코드나 문서를 작성했다는 뜻만이 아니다. 무엇을 바꿨는지, 어떻게 검증했는지, 사용자가 확인했는지, 남은 위험이 무엇인지 다음 참여자가 찾을 수 있어야 한다. [Task 종료 및 산출물 정책](../docs/12-task-completion-policy.md)은 이 정보를 같은 기준으로 남기기 위한 규칙이다.

## 2. Task 시작 시 해야 할 일

1. Product Roadmap과 관련 Task 문서를 읽는다.
2. 현재 branch, HEAD와 working tree가 예상과 같은지 확인한다.
3. 이미 진행 중인 같은 목적의 branch나 PR이 있는지 확인한다.
4. 이번 Task가 바꿀 것과 바꾸지 않을 것을 적는다.
5. 사용자 확인이 필요한 화면, 데이터 변경, 외부 발송을 미리 구분한다.

예상과 다른 기존 작업이나 사용자 변경이 있으면 임의로 지우거나 합치지 말고 먼저 비교·보고한다.

## 3. Task 진행 중 기록할 것

- 실제 변경 파일과 변경 이유
- 중요한 기술적 결정과 검토한 대안
- 실패하거나 폐기한 접근과 이유
- 실행한 테스트, 실패 원인과 수정 결과
- 실행하지 않은 검증과 이유
- 개인정보·secret 처리 방법
- 남은 문제와 후속 Task

코드 전체 diff를 문서에 복사할 필요는 없다. repository와 Git diff가 코드의 source of truth이고, 문서는 결정과 검증을 이해할 수 있게 설명한다.

## 4. Task 종료 전 확인할 것

1. 실제 변경이 Task 범위와 일치하는지 확인한다.
2. 변경 유형에 맞는 자동 테스트를 실행한다.
3. Finding을 P0/P1/P2/P3로 분류한다.
4. 5종 산출물의 경로와 상태를 기록한다.
5. Roadmap을 실제 결과에 맞게 갱신한다.
6. 개인정보, secret과 깨진 link를 검사한다.
7. 사용자 검수 상태를 사실대로 표시한다.
8. 허용된 파일만 stage한 뒤 cached 목록을 확인한다.

## 5. 5종 산출물 작성 방법

| 산출물 | 무엇을 기록하는가 |
| --- | --- |
| Implementation report | 실제 변경, 결정, 테스트, 미실행, 위험, 복구와 후속 작업 |
| SOP | 운영자·개발자가 반복해서 수행할 절차와 중단 조건 |
| User manual | 비개발 참여자도 이해할 수 있는 사용·확인 방법 |
| Roadmap update | 현재 구현 상태, 후속 Task, 추적 대상과 결정 이력 |
| User validation checklist | 자동 검증과 사용자가 직접 확인할 항목 및 상태 |

항상 5개 파일을 만들 필요는 없다. 다른 문서의 명확한 section에 포함했다면 그 위치를 link한다. 정말 해당하지 않으면 `N/A`와 구체적인 이유를 적는다. 아무 기록이 없는 것은 N/A가 아니다.

## 6. Checklist 작성과 완료 표시의 차이

체크박스를 만들면 상태는 `Checklist 작성됨`이다. 자동 명령이 모두 성공하면 `자동 검증 완료`로 올릴 수 있다. 사용자가 직접 확인해야 하는 항목이 남으면 `사용자 검수 대기`다.

사용자가 확인하고 승인한 뒤에만 `사용자 검수 완료`로 표시한다. 사용자가 문제를 발견하면 `사용자 검수 실패`로 기록하고 Finding을 만든다. 정말 검수 대상이 아닐 때만 이유와 함께 `적용 대상 아님`을 사용한다.

## 7. Finding 등급의 의미

- P0: 데이터 손실, 보안 사고 또는 시스템 중단 위험. 완료·게시·merge 금지.
- P1: 핵심 업무 흐름을 수행할 수 없음. 완료·게시·merge 금지.
- P2: 다음 단계 전에 고쳐야 할 구조·품질 문제. 기본적으로 수정한다.
- P3: 후속 개선이 가능한 문제. Task 또는 backlog에 연결한다.

P2를 예외적으로 수용하려면 사용자 승인, 책임자, 영향, 완화책, 재검토 시점과 후속 Task ID가 모두 필요하다.

## 8. PR을 만들면 안 되는 경우

- actual secret이나 개인정보가 current checkout에 남아 있음
- P0/P1 또는 승인되지 않은 P2가 있음
- runtime/dependency/migration 등 Task 범위 밖 변경이 섞임
- 필수 문서 link가 깨짐
- staged 파일이 allowlist와 다름
- 기존 정책과 양립할 수 없는 충돌이 해결되지 않음

사용자 검수 대기 상태에서 검토가 필요하면 draft PR로 만들고, 완료된 것처럼 표시하지 않는다.

## 9. 개인정보 작성 예시

허용:

- `검수 사용자 A`
- `시스템 관리자 검수자`
- `user@example.test`
- 마스킹된 식별값

금지:

- 실제 사용자 이름, 회사 이메일 또는 UPN
- 실제 tenant/client/object id
- password, token, webhook URL, Authorization header
- 고객·프로젝트의 민감한 원문

## 10. 자주 하는 실수

- checklist 파일이 있다는 이유로 사용자 검수 완료라고 표시함
- 실행하지 않은 UAT를 구현 완료와 같은 의미로 씀
- 작은 Task라며 SOP/User manual 상태를 누락함
- 기존 Roadmap의 역사적 결정을 삭제함
- `git add .`로 범위 밖 파일까지 stage함
- 실패한 테스트를 재실행 결과만 남겨 숨김
- 실제 사용자의 이름이나 이메일을 검수 증빙에 복사함

## 11. 간단한 예시

문서 전용 Task라면 implementation report에 변경한 정책, link 검사, PII/secret 검사와 미실행 runtime test 이유를 기록한다. SOP와 user manual이 별도 파일이 아니라 report section으로 충분하다면 각각 그 section을 link한다. 사용자 검수 항목이 남아 있으면 상태는 `사용자 검수 대기`이고 draft PR까지만 진행한다.

## 12. FAQ

### Q. 5종 산출물을 항상 5개 파일로 만들어야 하나요?

아니다. 각 산출물의 내용과 상태를 명확히 찾을 수 있으면 한 문서의 분리된 section도 가능하다.

### Q. 자동 테스트가 모두 성공하면 Task가 완료되나요?

아니다. 사용자 검수가 필요한 Task는 사용자 상태와 Finding gate도 통과해야 한다.

### Q. 사용자 검수 전에 PR을 만들 수 있나요?

검토가 필요하고 다른 gate를 통과했다면 `사용자 검수 대기`를 명시한 draft PR은 가능하다. merge 또는 Task 완료로 간주하지 않는다.

### Q. P2를 다음 Task로 넘길 수 있나요?

원칙은 현재 수정이다. 예외 수용에는 사용자 승인과 영향·완화책·후속 Task ID 등 canonical 정책의 모든 기록이 필요하다.

### Q. 과거 Git history에 개인정보가 있으면 어떻게 하나요?

current checkout 비식별화와 history rewrite는 다른 결정이다. 원문을 보고서에 복사하지 말고 영향 commit/file 수와 공개 범위를 확인한 뒤 별도 승인을 받는다.

### Q. merge 후 기존 branch를 지워도 되나요?

자동 삭제하지 않는다. 보존 필요성과 사용자 승인을 확인한 뒤 local/remote branch 정리를 별도로 수행한다.
