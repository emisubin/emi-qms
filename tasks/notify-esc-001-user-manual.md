# TASK-NOTIFY-ESC-001 User Manual

## 1. 무엇이 바뀌나요?

기한 알림을 확인할 업무가 100건을 넘더라도 앞쪽 업무만 반복 확인하지 않고 뒤쪽 업무까지 차례로 평가한다.

## 2. 왜 필요한가요?

기존에는 처음 선택된 100건이 다음 확인 때도 다시 앞에 와서 101번째 이후 업무가 오래 기다릴 수 있었다.

## 3. 화면이 바뀌나요?

아니다. API와 사용자 화면은 바뀌지 않는다.

## 4. 알림 날짜 정책이 바뀌나요?

아니다. L0~L3 예정일과 영업일 계산은 기존과 같다.

## 5. 수신자가 바뀌나요?

아니다. 담당자와 단계별 fallback 정책은 그대로다.

## 6. 100건 제한이 없어지나요?

한 번에 처리하는 최대 100건은 유지한다. 다음 poll이 가장 오래 평가되지 않은 업무를 이어서 확인한다.

## 7. 한 업무에 오류가 있으면 어떻게 되나요?

그 업무의 오류는 집계되고 같은 poll의 다른 업무는 계속 확인한다. 취소 요청은 즉시 멈춘다.

## 8. 중복 알림이 생기나요?

기존 escalation, notification, recipient와 delivery 중복 방지 규칙을 유지한다. 동시 evaluator 검증에서도 중복은 0이었다.

## 9. 서버가 재시작되면 처음부터인가요?

아니다. DB에 이미 기록되던 실제 평가 시각을 사용하므로 service가 다시 구성돼도 순환을 이어간다.

## 10. 전달 보장 수준

외부 전달은 at-least-once다. Exactly-once를 보장하지 않으며 이 Task는 provider 처리 방식을 바꾸지 않는다.

## 11. 사용자가 해야 할 작업

화면에서 새 설정을 할 필요는 없다. 검수자는 기존 기한 정책과 대량 후보의 순차 진행 의미를 확인한다.

## 12. 운영자에게 문의할 상황

- 같은 업무 알림이 중복돼 보임
- 뒤쪽 업무가 여러 poll 뒤에도 평가되지 않음
- 후보 실패 count가 반복 증가함
- 기존 L0~L3 시점이나 수신자가 달라짐

## 13. Persistent UAT 적용 상태

이번 Draft PR은 코드와 문서 검증 단계다. Persistent UAT worker는 계속 disabled이며 적용은 별도 controlled UAT 승인이 필요하다.

## 14. FAQ

### 기존 알림 기록을 삭제하나요?

아니다. 기존 history를 공정 순환 기준으로 읽을 뿐 삭제하지 않는다.

### 새 database migration이 필요한가요?

아니다. 기존 schema와 evaluation timestamp를 재사용한다.

### 실제 알림을 시험 발송했나요?

아니다. Isolated fake/no-op 환경에서 provider call 0으로 검증했다.
