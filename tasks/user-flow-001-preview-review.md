## 2. 한 줄 결론

현재 문서는 EMI의 전체 범위와 의존성을 파악하는 재료로는 유용하지만, 개인 개발 계획 자료에 canonical 계약·교육 문서·전수 갱신 의무까지 결합해 과도하므로, **“무엇을 왜 다음에 만들 것인가”를 고르는 비정규 개인 의사결정 지도**로 축소해 사용하는 것이 가장 적합하다.

## 3. 목적 적합성

문서가 해결하려는 문제는 “다음에 무엇을 개발해야 하는지 명확하지 않다”는 것이다. 이에 대해서는 다음 부분이 실제로 도움이 된다.

- 현재 구현, 부분 구현, 미구현 기능을 한눈에 구분한다.
- 18단계 업무 흐름과 후속 Task의 의존성을 연결한다.
- Pending List, 상태 집계, 자재, 검사, 제조, 물류, 정산이 어떤 순서로 이어지는지 보여준다.
- 품질을 IQC·LQC·OQC·전진검수/FAT으로 나눠 누락 가능성을 낮춘다.
- 각 기능에서 아직 결정하면 안 되는 정책을 후속 Task로 위임한다.

반면 문서의 상당 부분은 “무엇을 먼저 만들지”보다 현재 route·권한·알림·관리자 기능을 정확하게 설명하는 데 사용된다. 이는 기술 대조 문서나 온보딩 문서에는 유용하지만, 개인 개발 방향을 판단하는 핵심 질문에는 직접 답하지 못한다.

개인 계획 자료가 우선 답해야 할 것은 다음 다섯 가지다.

1. 지금 가장 큰 사용자 문제는 무엇인가.
2. 그 문제를 해결하는 가장 작은 end-to-end 기능은 무엇인가.
3. 선행 데이터·정책·보안 결정은 무엇인가.
4. 이번에는 무엇을 만들지 않을 것인가.
5. 구현 후 어떤 결과를 보면 다음 단계로 갈 수 있는가.

현재 문서는 3번과 4번은 잘 다루지만, 1번의 상대적 우선순위와 2번의 최소 vertical slice, 5번의 제품 성공 기준이 부족하다. 따라서 목적 적합성은 **부분 충족**이며, 방향 선택용 문서로 쓰려면 설명서보다 우선순위·의존성·성공 신호를 중심으로 재해석해야 한다.

## 4. 잘된 기획

- **현재와 목표 상태를 구분했다.** 미구현 기능을 존재하는 것처럼 표현하지 않고 도입 Task와 연결한 점은 향후 범위 오판을 줄인다.

- **업무 인수인계를 중심에 뒀다.** EMI의 핵심 가치는 부서별 화면 수가 아니라 단계 완료 후 다음 담당자의 내 업무와 알림이 이어지는 데 있다. 문서가 이 축을 일관되게 유지한다.

- **18단계와 역할별 관점을 연결했다.** 특히 품질 여정을 검사 단계별로 나눈 것은 기능 누락과 잘못된 담당자 연결을 발견하는 데 유용하다.

- **구현과 미구현 구간의 상세도를 다르게 했다.** 미구현 검사 양식·첨부·포장 정책을 확정하지 않고 후속 planning으로 넘긴 판단은 Roadmap과 충돌하지 않는다.

- **공통 진입점 세 가지가 실용적이다.** 내 업무, 알림 deep link, 메뉴 직접 진입은 향후 화면마다 다시 논의할 필요가 없는 유용한 UX 패턴이다.

- **후속 Task 역색인이 좋다.** `TASK-007A`부터 물류·정산까지 무엇이 선행되고 어떤 결정을 위임했는지 확인할 수 있어 개인 개발 순서를 판단하는 데 직접 도움이 된다.

- **권한과 알림을 새로 정의하지 않았다.** Backend 정책과 Roadmap의 채널 계약을 보존함으로써 이 문서가 범위를 무단 확장하지 않는다.

- **생산계획 완료 시 설계와 구매가 함께 활성화되는 현재 동작을 표준 단계 전이와 구분했다.** 단순한 직렬 흐름으로 오해할 수 있는 부분을 비교적 잘 설명했다.

이 장점들은 코드와 일치하기 때문만이 아니라, 향후 기능의 책임 경계와 의존성을 이해하는 데 도움이 되기 때문에 유지할 가치가 있다.

## 5. 과도하거나 불필요한 범위

### Canonical 선언

개인 검토용 문서에는 필요하지 않다. Product Roadmap이 이미 제품 방향과 Task 순서의 source of truth이고, 각 기능은 별도 interview·planning·review를 거친다. `docs/13`까지 전체 유저플로우의 canonical source로 선언하면 두 문서가 동시에 최신 상태여야 하며 충돌 판정 비용이 증가한다.

권장안은 `docs/13`을 “개인 개발 판단용 참고 기준선”으로 두고, 충돌 시 Roadmap과 승인된 최신 Task 계약을 따른다고 명시하는 것이다.

### 사용자 온보딩·교육 목적

현재는 대부분의 검사·제조·물류·정산 화면이 미구현이므로 너무 이르다. 예정 흐름을 교육 자료로 사용하면 실제 UI가 확정될 때마다 다시 작성해야 하고 사용자가 목표 상태를 현재 기능으로 오해할 수 있다. 운영 전환 또는 해당 역할의 UAT 완료 뒤 작성하는 편이 낫다.

### `docs/02`·`docs/04` Phase B 정렬

두 문서가 현재 18단계와 권한 상태보다 오래된 것은 사실이다. 그러나 이를 개인 계획 문서 사용의 선행조건으로 묶을 필요는 없다.

- `docs/13`을 게시 가능한 canonical 문서로 만들려면 정렬이 필요하다.
- 개인 참고 문서로 유지한다면 Phase B는 현재 Task에서 불필요하다.
- 기존 두 문서의 drift는 필요 시 별도 `DOCS_GOVERNANCE` 범위로 처리하는 것이 책임이 명확하다.

따라서 현재 사용자 의도에서는 **Phase B를 보류**하는 것이 적합하다.

### 모든 후속 Task의 의무 갱신 규칙

제거하는 것이 좋다. 화면 문구나 작은 action 변경까지 모두 `docs/13` 갱신을 요구하면 1인 개발에서 문서가 병목이 된다.

대신 다음과 같은 사건 기반 규칙이면 충분하다.

- 18단계 순서·담당 역할·완료 조건 변경
- 최상위 내비게이션 변경
- 공통 진입점이나 handoff 방식 변경
- 프로젝트·구매품목·패널 등 핵심 업무 단위 변경
- 주요 기능군 하나가 실제 UAT를 통과한 시점

그 외 세부 동작은 Roadmap과 해당 Task 문서가 소유해야 한다.

### 13개 여정의 전수 상세 설명

13개 역할·검사 단계의 coverage 자체는 유용하지만, 모든 여정을 상세 온보딩 문장으로 유지할 필요는 없다. 개인 계획 자료에는 `역할 × 단계 × 현재 상태 × 다음 담당자 × 도입 Task` 표가 더 빠르고 유지비도 낮다. 상세 여정은 해당 기능을 실제로 구현할 때 작성하면 된다.

### 구현 route·권한·알림 세부의 반복

정확한 route, 관리자 기능 목록, 알림 채널 운영 상세는 현재 상태를 증명하는 데는 유용하지만 개발 우선순위 판단에는 비중이 과하다. Roadmap·권한 문서를 링크하고 이 문서에는 “현재 진입 가능/불가” 정도만 남기는 편이 낫다.

### 목표 메뉴 위치의 조기 확정

“부서 업무” 묶음과 Pending List의 내 업무·프로젝트 하위 고정은 아직 검증되지 않은 UI 결정이다. 특히 생산관리 담당자가 전체 Pending을 관리해야 하므로 Pending 전용 workspace가 필요할 가능성이 있다. `TASK-007A` 사용자 흐름을 확인하기 전에는 위치를 확정하지 말고 후보로 두는 편이 안전하다.

## 6. 빠진 기능·관점

### 업무 단위와 병렬 진행

가장 중요한 누락이다. 현재 마스터 flow는 프로젝트 하나가 1→18로 이동하는 직렬 흐름처럼 보이지만 실제 EMI는 다음 단위가 섞인다.

- 프로젝트 단위: 초기 생산계획·담당자
- 구매품목 단위: 발주·도착·IQC·입고 확정
- 패널 단위: 키팅·제조·검사·납품
- 검사·포장 단위: 결과와 증빙

일부 구매품목이나 패널만 다음 단계로 진행할 수 있으므로, “어떤 단위가 언제 병렬로 전진하고 무엇이 전체 완료를 막는가”가 별도 map으로 필요하다. 이는 화면보다 데이터 모델과 Task 분할을 결정하는 핵심 관점이다.

### 최소 vertical slice

전체 기능 목록은 있지만 가장 작은 유효 제품 단위가 없다. 권장 첫 slice는 다음과 같다.

`구매품목 도착 → IQC 판정 → 부적합 Pending 또는 입고 확정 → 패널 키팅 → 제조 내 업무 생성`

이 slice는 자재·품질·Pending·handoff를 실제로 연결해 EMI의 핵심 가치를 검증할 수 있다.

### 진행 중 업무의 재배정

담당자 fallback은 업무 생성 시점의 누락을 해결하지만, 퇴사·부서이동·휴직 후 이미 생성된 미완료 업무를 누가 인수하는지는 해결하지 않는다. 운영 전에 재배정 권한·이력·알림 정책이 필요하다.

### 완료 기록의 정정·재개

단계는 후퇴하지 않는다는 원칙은 좋지만, 잘못 완료한 기록을 어떻게 정정하고 누가 재개를 승인하는지가 없다. 검사와 납품 기능 전에 append-only 정정, 재검사, reopen 권한을 정해야 한다.

### 임시 저장·부분 완료·재시도

모바일 검사·사진·제조 작업은 네트워크나 첨부 업로드 실패가 발생할 수 있다. 초안 저장, 이어하기, 중복 제출 방지, 일부 패널 일괄 처리 실패 시 복구 방식이 후속 planning의 공통 질문으로 필요하다.

### 첨부 storage 운영 계약

Pending·IQC·물류·모바일의 공통 선행조건이다. 저장 위치, 파일 크기·형식, 악성 파일 방어, 보존·백업·복구, 접근권한을 기능별로 반복 설계하지 않도록 먼저 최소 계약을 정해야 한다.

### 제품 성공 신호

문서 완성 여부가 아니라 다음과 같은 업무 결과를 측정해야 한다.

- 다음 담당자에게 전달되지 않은 완료 건수
- 직접 입력 화면으로 연결되지 않는 내 업무 비율
- 장기 미조치 Pending 건수와 평균 체류시간
- 필수 증빙 누락으로 되돌아간 작업 수
- 프로젝트 병목을 찾는 데 걸리는 시간

이 중 2~3개만 있어도 다음 기능 우선순위를 훨씬 쉽게 판단할 수 있다.

## 7. 개발 방향 충돌·위험

- **Roadmap과 별도 canonical 문서의 이중화 위험이 있다.** 제품 방향은 Roadmap, 상세 흐름은 `docs/13`, 실제 권한은 Backend로 분산되면 작은 변경도 세 곳의 동기화를 요구한다.

- **Change 002의 개인 검토용 목적과 현재 Roadmap의 Phase A→Phase B 게시 흐름이 어긋난다.** 이번 review에서는 더 최신 사용자 정정인 Change 002를 적용해야 한다. 따라서 Phase B나 게시를 자동 다음 단계로 보지 않는다.

- **직렬 18단계 도식이 실제 병렬성을 가릴 수 있다.** 이 상태로 기능을 구현하면 프로젝트 전체 stage와 구매품목·패널 상태를 잘못 결합할 위험이 있다.

- **Pending 위치 결정이 너무 이르다.** 내 업무·프로젝트의 contextual 진입은 필요하지만, 생산관리의 전체 이슈 관리 workspace까지 막아서는 안 된다.

- **Home이 핵심 업무 데이터보다 먼저 오면 빈 dashboard가 될 수 있다.** 병목 집계와 자재·검사 데이터가 충분해진 뒤 구현해야 사용자 가치가 생긴다.

- **모바일을 하나의 큰 Task로 다루면 범위가 커진다.** 반응형 내비게이션·overflow 방지는 모든 Task에서 유지하되, 사진 압축·업로드 재시도 같은 기능은 실제 IQC·제조 흐름과 함께 구현하는 편이 낫다.

- **알림 고도화가 핵심 업무 흐름보다 앞서갈 위험이 있다.** 현재 외부 delivery 기반은 충분히 구축돼 있으므로 자동 event coverage·사용자 preference보다 Pending·자재·검사·제조를 먼저 완성하는 것이 제품 가치가 높다.

현재 read-only 검토에서 실제 구현을 즉시 차단하는 P0/P1/P2는 확인되지 않았다. 기존 `docs/02`·`docs/04` 불일치는 `docs/13`을 canonical 게시물로 만들 때만 게시 차단 조건이며, 개인 계획 자료 사용을 막지는 않는다.

## 9. 권장 개발 순서(Now/Next/Later)

### Now

1. `docs/13`을 canonical 문서가 아닌 개인 개발 판단 자료로 취급한다. Phase B·게시·전수 갱신은 진행하지 않는다.
2. 프로젝트·구매품목·패널·검사 단위의 병렬 진행 관계를 한 장의 dependency map으로 판단 기준에 추가한다.
3. `TASK-007A` Pending List MVP를 먼저 기획한다.
   - 이슈 생성, 조치 담당, 상태, 코멘트, 재검사·종결을 핵심으로 한다.
   - 첨부 storage 정책이 오래 걸리면 text-first UAT slice와 운영용 증빙 완성 단계를 분리한다.
4. 신규 화면에 필요한 최소 Action Feedback 계약을 Pending 구현과 함께 적용한다. 별도 대형 UX 개편으로 만들 필요는 없다.
5. Pending이 안정되면 `TASK-007B` 병목 상태 집계를 구현한다.
6. 병행 정책 결정으로 미완료 업무 재배정, 완료 기록 정정·재개, 첨부 storage 경계를 확정한다.

### Next

2. `TASK-009A` IQC·사진·부적합 Pending·PDF 기반
3. `TASK-010A` 패널별 키팅·제조 내 업무 생성
4. 위 기능과 함께 모바일 최소 기반을 적용한다.
   - 동일 URL
   - 390px overflow 0
   - 현장 핵심 action
   - 사진 업로드 실패·재시도
5. `TASK-011A` 제조 작업·중단
6. `TASK-012A` LQC·OQC·전진검수·FAT
7. 실제 집계 데이터가 충분해진 뒤 Home MVP를 만든다.

현재 Roadmap의 Home 선행 순서는 뒤로 미루고, `TASK-MOBILE-001`은 “공통 responsive 기반”과 “사진·현장 입력 능력”으로 나눠 실제 업무 Task에 맞춰 적용하는 방향을 권장한다. 실제 순서 변경은 별도 Roadmap 승인 대상이다.

## 10. 최종 권고

`docs/13-user-flow-baseline.md`는 폐기할 문서가 아니다. 다만 지금 상태 그대로 canonical 제품 계약이나 교육 문서로 승격하지 말고, 다음 세 부분만 개인 개발 판단의 중심으로 사용하는 것이 좋다.

- 현재 대 목표 기능 gap
- 18단계 handoff와 업무 단위 의존성
- 후속 Task 매핑과 Now/Next/Later 우선순위

Canonical 선언, Phase B 의무, 모든 후속 Task의 동시 갱신, 전수 역할 온보딩 설명은 제거하거나 보류하는 것이 1인 개발에 적합하다. EMI의 다음 제품 가치는 Home·디자인·알림 고도화보다 **Pending → 병목 집계 → 자재 도착 → IQC → 키팅 → 제조 handoff**의 첫 end-to-end 업무 slice에서 나온다.

이번 review로 기획 작성 흐름을 종료한다. Fable 재작성이나 추가 review round는 권고하지 않으며, Repository 파일·runtime·Git 상태는 수정하지 않았다. 게시·commit·PR·merge는 이번 read-only 검토의 대상이 아니다.
