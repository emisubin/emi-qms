# TASK-NOTIFY-004 User Manual — 발송 실패 처리

## 1. 화면 변경 여부

이번 Task의 사용자 화면 변경은 없다. 현재 관리자 알림 발송 상태 화면의 Pending 재발송, Failed 확인·제외와 attempt 상세를 그대로 사용한다.

## 2. Pending과 Failed의 차이

- Pending은 아직 worker가 처리할 수 있는 대기 상태다.
- Failed는 자동 재시도할 수 없는 오류이거나 정해진 재시도 횟수를 모두 사용한 최종 상태다.

선택 재발송 버튼은 Pending의 처리 시각을 앞당기는 기능이다. Failed를 다시 발송하는 버튼이 아니다.

## 3. Failed가 보이면 할 일

1. 해당 항목의 오류 분류와 시도 이력을 확인한다.
2. 설정·권한·수신 대상 문제라면 운영 담당자에게 알린다.
3. 확인이 끝나면 확인 처리하거나 목록 제외 처리한다.
4. 같은 알림을 꼭 다시 보내야 한다면 Failed row를 직접 바꾸지 말고 별도 기능 요청을 등록한다.

## 4. 왜 Failed 재발송 버튼이 없나요?

일부 외부 채널은 전송에 성공했어도 시스템이 결과를 저장하기 전에 중단될 수 있다. 이 경우 다시 보내면 같은 알림이 중복 전달될 수 있다.

또한 기존 자동 재시도 횟수와 시도 이력을 그대로 보존해야 한다. 안전한 수동 재처리에는 별도 감사 이력과 중복 위험 확인 정책이 필요하므로 현재 P2 범위에는 포함하지 않았다.

## 5. 확인·제외의 의미

확인과 목록 제외는 운영자가 해당 실패를 검토했다는 상태만 기록한다. 실제 발송 결과를 Sent로 바꾸거나 새 발송을 만들지 않는다.

## 6. 보장 수준

시스템은 정상 worker 경쟁에서 같은 delivery의 동시 provider 호출을 claim/lease로 막는다. 외부 provider와 DB를 하나의 transaction으로 묶을 수 없으므로 전체 전달 보장은 at-least-once이며 exactly-once가 아니다.

## 7. 하면 안 되는 작업

- Failed 상태를 DB에서 Pending 또는 Sent로 직접 변경
- Attempt count나 attempt history 삭제
- 같은 알림을 확인 없이 반복 생성·발송
- Recipient, 제목·본문 또는 provider 응답을 일반 보고서에 복사

## 8. 향후 기능 요청

Failed 수동 재처리가 실제 운영에 필요하면 다음 내용을 식별자 없이 전달한다.

- 어떤 채널인지
- 어떤 실패 분류인지
- 자동 retry를 모두 사용했는지
- 외부 호출이 시작됐는지
- 중복 가능성을 사용자가 확인할 수 있는지

이 요청은 별도 신규 기능 planning과 사용자 승인을 거친다.

## 9. 사용자 검수 체크리스트

- [ ] Pending과 Failed의 차이를 이해
- [ ] 선택 재발송이 Pending 전용임을 확인
- [ ] Failed 확인·제외가 발송 결과를 변경하지 않음을 확인
- [ ] Failed 재발송이 현재 제공되지 않는 이유를 이해
- [ ] At-least-once와 중복 가능성을 이해
- [ ] 별도 신규 기능 요청 절차를 확인
